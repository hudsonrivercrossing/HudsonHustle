# V2.2 System Player PRD

## Product Summary

`v2.2` begins Hudson Hustle's first non-human controller milestone.

The first step is a built-in server-side deterministic system player that can occupy a seat in the existing authoritative multiplayer architecture. It should behave like a normal player from the room and rules perspective, while remaining operationally simple, testable, and fair.

## Problem

Hudson Hustle `v2.1` now has a stable multiplayer room flow, authoritative backend, reconnect model, and shell/system foundation. But there is still no system-owned player that can:
- fill an empty seat for testing or lightweight solo/mixed play
- validate controller abstractions before external or assistive agent work
- exercise real game progression without requiring a full human table

Without a first-party system player:
- multiplayer testing still depends too heavily on multiple humans or manual multi-browser orchestration
- seat/controller architecture remains partially theoretical
- later `bot`, `agent`, and `human+agent` work risks being shaped without a proven first controller path

## Goals

- Add the first system-owned player as a real seat controller inside the existing authoritative backend.
- Keep the first controller deterministic or seed-controlled enough to debug and test reliably.
- Ensure the system player uses the same legal action pathway as a human seat.
- Preserve hidden-information boundaries and fairness.
- Establish a stable controller abstraction that later `bot`, `agent`, and `human+agent` work can extend.

## Non-Goals

- No external LLM or hosted model dependency.
- No user-provided autonomous agents.
- No chat-first assistant interface.
- No public “coach” or `human+agent` collaboration flow in this first PRD.
- No redesign of map, scoring, or core gameplay rules.
- No attempt to ship a highly sophisticated or “human-like” opponent immediately.

## User Stories

1. As a developer, I want to start a room with a built-in system player so I can exercise real multiplayer flows and game progression without needing a second human.
2. As a future designer of mixed-controller play, I want the first system player to follow the same seat/action contracts as a human so later controller types can reuse the same architecture.
3. As a player, I want a system player to take legal turns predictably and fairly so the game remains understandable and trustworthy.

## Solution

Hudson Hustle `v2.2` should introduce a first-class built-in system player that is owned by the server and represented as a seat controller type.

The system player should:
- be assignable to a seat through room setup or room creation flow
- receive only the public and private information that a normal seat should see
- choose legal actions through a deterministic decision adapter
- submit normal action intents through the authoritative server path
- obey turn timing and turn progression rules like any other seat

The first version should prioritize:
- legality
- determinism
- debuggability
- operational simplicity

It should not prioritize:
- advanced personality
- explanation UX
- natural-language interaction
- external extensibility

The product shape should remain clearly “Hudson Hustle with an internal bot seat,” not “Hudson Hustle with general agents.”

## Product Decisions

The first `v2.2` PRD locks these decisions:

1. The first system player should be available in normal room setup.
- It should not remain hidden behind a dev-only switch.
- The product value of `v2.2` is that a normal player can start a room with one or more built-in system seats.
- The UI should still make clear that this is a `bot` seat, not a human seat and not a future `agent`/`human+agent` mode.

2. The first release should support any number of bot seats up to room size.
- A room may mix human and bot seats.
- The architecture should not special-case “exactly one bot only.”
- This keeps the seat/controller model honest and avoids shipping a fake abstraction that immediately needs rewriting.

3. The first bot should target a competent baseline, not explicit easy/hard tiers.
- `v2.2` should not introduce difficulty levels yet.
- Instead, the first milestone should ship one deterministic baseline policy that is:
  - legal
  - understandable
  - not obviously self-destructive
  - good enough to complete full games credibly
- The bot may and should borrow from current Hudson Hustle strategy heuristics and “how to win” guidance where that improves move quality.
- Difficulty tiers can come later once the baseline controller is stable and measurable.

## Scope

### In Scope

- Introduce a server-owned non-human controller path for one seat type.
- Add the minimum room and seat modeling needed to represent a system player.
- Allow rooms to use any mix of human and bot seats up to room size.
- Allow the system player to take legal turns through the authoritative action path.
- Define the first decision policy as deterministic or seed-controlled.
- Add the minimum setup UX needed to place one or more bot seats in a normal room setup flow.
- Add tests that prove legality, hidden-information isolation, and stable turn progression.
- Update docs for controller terminology and `v2.2` system-player direction.

### Out Of Scope

- External model-driven agents.
- User-authored plugins or CLI controller protocols.
- Rich difficulty settings such as `easy` / `medium` / `hard`.
- `human+agent` suggestion UX.
- Conversation, explanation, or coaching surfaces.
- Broad matchmaking, solo campaign, or ranking features.

## Implementation Decisions

- The first system player should be a built-in server-side controller, not a client-owned controller.
- It should be available through the normal room setup path, not only through a hidden dev surface.
- It should run through the existing authoritative room/game flow instead of bypassing validation.
- The decision layer should be implemented as a deterministic decision adapter over shared rules and game state.
- The controller abstraction should stay seat-oriented, so later controller modes can reuse room and game wiring.
- The first controller should consume only the same public/private view a normal seat should have.
- The first release should support any number of bot seats up to room size, rather than hard-coding a single bot-seat limit.
- The first release should ship one competent baseline policy before introducing difficulty tiers.
- The baseline policy may reuse or formalize existing Hudson Hustle strategy heuristics where that improves move quality.
- The implementation should preserve:
  - `V1` local pass-and-play
  - `V2.1` human multiplayer
- The first `v2.2` controller milestone should still treat:
  - `bot`
  - `agent`
  - `human+agent`
  as distinct concepts, even if only the first one is implemented now.

## Testing Decisions

- Add deterministic server tests for:
  - legal action selection
  - turn completion
  - hidden-information boundaries
  - seat/controller lifecycle
  - mixed human/bot seat compositions
- Prefer behavior-first tests over internal implementation assertions.
- Add at least one browser or multiplayer smoke path if setup UX changes.
- Verify that human-only rooms still behave exactly as before.
- If seeds or deterministic policy controls are introduced, test them explicitly.

## Open Questions

- How should normal room setup express controller selection without making the lobby feel heavier than `v2.1`?
- Which strategy heuristics should be considered mandatory for the baseline bot before the first release candidate?
- Should the first implementation expose deterministic seed control only for testing/dev, or also in normal product surfaces?
