# V2.1 Design Lock

## Purpose

This document locks the minimum design-system decisions needed before `v2.1` starts a real implementation slice.

It is intentionally short.

## Locked Decisions

### 1. Typography Direction
- Hudson Hustle should move toward a stronger editorial shell voice.
- Use a more distinctive display direction for:
  - page titles
  - section titles
  - major status statements
- Keep body and operational UI text calm, practical, and highly legible.
- Do not use expressive display typography for dense gameplay data.
- `v2.1` active font pair:
  - display: `Fraunces`
  - body / operational UI: `Inter`
- Roles:
  - `Fraunces` is for shell identity, status headlines, page titles, and section titles.
  - `Inter` is for controls, timers, labels, chips, panel copy, and gameplay-adjacent operational text.
- Backups and boundaries:
  - primary sans backup: `Space Grotesk + IBM Plex Sans`
  - high-intensity brand experiment: `Archivo Black + DM Sans`
  - do not use `Balsamiq Sans` as the main shell font; keep it only as a playful boundary test

### 2. Shell Intensity
- `v2.1` should be shell-first and visibly more authored than `v2.0`.
- The shell should absorb the strongest influence from `ideago.cc`.
- The product should feel more editorial, spacious, and deliberate.
- It should not become fashion-minimal, fragile, or poster-first.

### 3. First Implementation Slice
- The first design-system implementation slice for `v2.1` is:
  - `status/banner system`
- This slice should cover:
  - turn status
  - waiting state
  - host/setup state
  - reconnect/failure state
  - timer treatment
- It should establish:
  - typography hierarchy
  - banner surface language
  - timer visual language
  - stronger shell identity

### 4. Map Boundary
- The map is not the first redesign target in `v2.1`.
- `v2.1` may influence map-adjacent language only through:
  - framing
  - surrounding shell composition
  - calmer typographic discipline
  - ownership badge integration
- `v2.1` should not:
  - reduce route readability
  - weaken color clarity
  - over-stylize labels
  - turn the board into an editorial poster surface

## Immediate Next Step
- implement the `status/banner system` slice using this lock
- validate it in:
  - setup / lobby
  - multiplayer in-game shell
  - local play shell where relevant

## Explicit Non-Goals
- no broad map restyle yet
- no full shell rewrite yet
- no giant component-library push before slice validation
