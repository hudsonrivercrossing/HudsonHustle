# V2.3 Slice 1 Implementation PRD

## Product Summary

This document scopes the first implementation slice of `v2.3` concise UI and game history work.

Slice 1 creates the first canonical online multiplayer game-history foundation:
- every completed multiplayer match should have a durable ordered move log
- the log should be detailed enough to review the game move-by-move and round-by-round
- the first storage model should use `action log + checkpoint snapshots`

This slice is a backend and persistence foundation slice, not a replay or analytics UI slice.

## Current State

`v2.2` already persists enough state to resume active rooms:
- `rooms`
- `room_seats`
- `game_snapshots` for the latest saved `GameState`

The current Postgres repository also writes a very thin `game_events` audit row with:
- `sequence`
- `event_type = "room_saved"`
- a minimal payload

That is not yet a real game-history system:
- it does not preserve each authoritative move as a readable match record
- it does not preserve sparse historical checkpoints for later reconstruction
- it is not defined as a stable logging contract for replay, review, or later bot-learning work

## Problem

`v2.3` needs a trustworthy historical record of each online multiplayer match.

Without that foundation:
- a future replay flow would have no stable move-by-move source
- future player-specific highlights would depend on ad hoc reconstruction
- future bot-skill generation from completed games would inherit ambiguous or incomplete data
- debugging completed games would still depend too much on final state instead of full progression

The product requirement is not “save one final score screen.” It is:
- keep a durable account of the game from start to finish
- record each move clearly enough that the full game can be reconstructed like a chess record
- do this without turning the whole runtime into a heavyweight new event-sourcing architecture

## Goals

- Store a canonical ordered history for every online multiplayer match created after this slice ships.
- Record every authoritative gameplay move with enough metadata to know:
  - who acted
  - what action happened
  - when in the match it happened
  - what state boundary it produced
- Add sparse immutable checkpoints so downstream consumers do not need to replay from move zero every time.
- Reuse the current persistence model where possible instead of replacing it wholesale.
- Preserve hidden-information safety by keeping full private state out of normal event rows.

## Non-Goals

- No replay UI.
- No player-facing match viewer.
- No AI-written post-game summaries.
- No bot-skill generation yet.
- No local pass-and-play history capture.
- No full migration to a pure event-sourced backend.
- No gameplay rule or scoring changes unless required to log authoritative actions correctly.

## User Stories

1. As a maintainer, I want each online match to leave behind a clear move-by-move record so I can inspect how the game actually unfolded.
2. As a future replay/highlight feature builder, I want a stable history contract that I can consume without reverse-engineering room persistence.
3. As a future bot-training workflow, I want completed matches to expose durable sequences, turn boundaries, and outcomes instead of only final scores.

## Solution

Slice 1 should introduce a canonical game-history contract built around two layers:

1. action log
- append one immutable history event for each authoritative gameplay transition after a room becomes active
- include enough metadata to reconstruct move order and turn/round structure

2. checkpoint snapshots
- store sparse immutable `GameState` checkpoints at stable boundaries
- use those checkpoints to make later reconstruction practical without saving a full snapshot after every move

The current live-resume snapshot table should remain in place for runtime room restore.

The first canonical contract should cover online multiplayer only and should begin at game start:
- `game_started`
- initial ticket confirmation sequence
- main-phase moves
- timeout-driven automatic actions
- server-owned `bot` actions
- final game completion

## Scope

### In Scope

- online multiplayer history only
- evolution of the current `game_events` table into a canonical gameplay history log
- a new immutable checkpoint table for historical snapshots
- server-side logging contract for:
  - human-submitted actions
  - server-owned `bot` actions
  - timeout-generated actions
  - game start
  - game finish
- deterministic tests for history completeness and ordering
- documentation of the history schema and logging contract

### Out Of Scope

- user-visible replay routes or pages
- analytics dashboards
- derived highlights
- bot-skill generation
- backfilling old completed games into the new canonical format

## Implementation Decisions

### 1. Keep the current runtime snapshot path

The existing `game_snapshots` table should remain the source of the latest saved room state for restore/reconnect behavior.

This slice should not repurpose it into a full historical archive.

### 2. Upgrade `game_events` into the canonical action log

The existing `game_events` table should stop acting like a thin `room_saved` audit sink and instead become the canonical immutable gameplay event log.

The log contract should be:
- one row per authoritative gameplay transition
- ordered by `room_code + sequence`
- sequence should align with `snapshotVersion`
- `game_started` should use sequence `0`
- later state-changing gameplay events should use the resulting post-transition `snapshotVersion`

Legacy `room_saved` rows from older games may remain in storage, but they are not part of the `v2.3` canonical history guarantee.

### 3. Add immutable checkpoint snapshots

Add a new `game_history_checkpoints` table rather than overloading the current latest-snapshot table.

Recommended schema direction:
- `room_code`
- `snapshot_version`
- `checkpoint_type`
- `snapshot`
- `created_at`

Primary key:
- `(room_code, snapshot_version)`

This keeps checkpoints aligned to the same monotonic version boundary used by the event log.

### 4. Use sparse checkpoint cadence, not full snapshot-per-move persistence

The first checkpoint policy should be deterministic and minimal.

Required checkpoints:
- after `game_started`
- after each completed main-phase turn handoff where the active seat changes
- after any phase transition into `gameOver`

Optional future checkpoint types can be added later, but downstream consumers should only rely on the required cadence above.

### 5. Define one stable history-event payload contract

Each canonical history event payload should include:
- `schemaVersion`
- `snapshotVersion`
- `phaseBefore`
- `phaseAfter`
- `activeSeatIdBefore`
- `activeSeatIdAfter`
- `roundNumber`
- `turnNumber`
- `turnActionIndex`
- `actor`
  - `seatId`
  - `playerId`
  - `controllerType`
  - `ownership`
- `source`
  - `human_request`
  - `server_bot`
  - `server_timeout`
  - `server_resume`
- `action`
  - the logical applied action payload
- `summary`
  - small reconstruction-oriented facts such as score deltas, route claim id, station city id, or ticket-keep counts when relevant

The payload should be rich enough for reconstruction, but it should not duplicate the entire `GameState`.

### 6. Keep hidden information out of normal event rows

Normal history events should not store full private hands or ticket lists directly in the per-move payload.

Private state should remain available only through:
- internal checkpoint snapshots
- future privileged/internal retrieval paths

This keeps the stored history compatible with possible future user-visible projections without first leaking all hidden information through the base event stream.

### 7. Represent timeout and bot actions explicitly

The history contract must make server-driven actions first-class, not invisible side effects.

That means:
- a `bot` move must appear with the acting bot seat and `source = server_bot`
- a timeout-driven automatic resolution must appear explicitly with `source = server_timeout`
- restore/resume-triggered bot continuation must still preserve the acting seat and resulting move order

### 8. Turn and round indexing must be explicit

The stored history should be queryable without replaying the whole game just to know where a move belongs.

So each event should include:
- `turnNumber`
- `turnActionIndex`
- `roundNumber`

Recommended semantics:
- `turnNumber`: monotonic count of started turns once the room enters the main gameplay loop
- `turnActionIndex`: `1..n` within that turn
- `roundNumber`: monotonic seat-cycle count during the main gameplay loop
- during `initialTickets`, `turnNumber` and `roundNumber` may be `null`

### 9. No backfill guarantee for older finished games

This slice should not block on trying to reconstruct old `v2.0` / `v2.2` games into the new contract.

The guarantee is:
- all new online multiplayer matches created after rollout use the canonical history model

## Schema Direction

### Existing Tables Kept

- `rooms`
- `room_seats`
- `game_snapshots`

### Existing Table Evolved

- `game_events`
  - keep table name for migration simplicity
  - treat it as the canonical immutable gameplay history log for `v2.3`+ matches

### New Table

- `game_history_checkpoints`
  - immutable sparse checkpoint snapshots keyed by `room_code + snapshot_version`

## Logging Contract

### Required Canonical Event Types

- `game_started`
- `action_applied`
- `game_finished`

This first version should avoid over-fragmenting event types.

Instead, special cases such as bot actions and timeout actions should be distinguished inside the required payload fields:
- `actor`
- `source`
- `action`

### When To Append Events

Append one canonical event when:
- a room becomes an active game
- an authoritative gameplay action is committed and changes state
- the game reaches `gameOver`

Do not append canonical history rows for:
- lobby-only seat assignment changes
- ready toggles
- websocket connect/disconnect noise
- generic room saves that do not represent gameplay progression

### Required Ordering Rule

Within one room:
- canonical history ordering is strictly increasing by `sequence`
- `sequence` must correspond to the post-transition `snapshotVersion`
- there must never be two canonical gameplay events with the same `room_code + sequence`

## Testing Decisions

- Add deterministic server tests proving that:
  - a completed online multiplayer game writes a full ordered history
  - human actions, bot actions, and timeout-driven actions all appear with the correct source/actor metadata
  - checkpoint rows are written at the required cadence
  - the history is sufficient to walk the game from `game_started` to `game_finished` move-by-move
- Add at least one mixed human/`bot` fixture so the history contract is not implicitly human-only.
- Keep current persistence and restore tests green to prove the new history path does not break runtime room resume.
