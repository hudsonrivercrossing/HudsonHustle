## Recommended Execution Order
1. persist a canonical multiplayer game history log
2. prove that saved history can be fetched and reconstructed reliably
3. simplify the live gameplay shell and remove obsolete chrome
4. clean up typography/color drift and align the shipped surface with the current design language
5. freeze docs and record what stays for `v2.4` and `v3`

## Issue 1

### Title
Persist a canonical ordered game history for every multiplayer match

### Objective
- ensure the authoritative backend stores enough ordered move data to reconstruct a full match after it ends

### Acceptance Criteria
- every completed multiplayer game has a durable ordered history
- history uses `action log + checkpoint snapshots`
- history records include acting seat, move type, turn/round sequencing, and endgame outcome anchors
- the stored data is clear enough to review a game move-by-move like a chess record
- history capture does not change game rules or authoritative validation behavior

### Validation
- deterministic server tests for complete-game history persistence

### Out Of Scope
- replay UI
- AI-generated summaries
- local pass-and-play history

## Issue 2

### Title
Expose a minimal retrievable match-history surface for review and future replay consumers

### Objective
- make the saved history usable instead of only stored

### Acceptance Criteria
- a game history can be fetched in order after game completion
- the returned data is sufficient for internal review of each move and round progression
- the contract is stable enough for future replay/highlight tooling
- the first retrieval surface stays internal or low-prominence

### Validation
- end-to-end server test for history fetch after a completed match

### Out Of Scope
- polished user-facing replay experience

## Issue 3

### Title
Simplify the live multiplayer UI shell and remove low-value interface noise

### Objective
- make the shipped `v2` surface more concise before larger `v2.4` UX work

### Acceptance Criteria
- clearly low-value or redundant UI sections are removed or reduced
- gameplay-critical information remains visible and easier to scan
- setup, lobby, in-game flow, and endgame all still pass smoke validation

### Validation
- browser smoke on setup, lobby, gameplay, and endgame
- one findings-first visual review

### Out Of Scope
- large-scale UX redesign
- new major panels or assistant surfaces

## Issue 4

### Title
Align gameplay typography, colors, and layout tokens with the current design direction

### Objective
- remove lingering legacy style drift so the product feels visually coherent

### Acceptance Criteria
- legacy font/color usage in shipped gameplay surfaces is replaced or normalized
- layout spacing and hierarchy feel tighter and more intentional
- style cleanup does not silently change gameplay behavior

### Validation
- visual review against current gameplay surfaces
- web build and browser smoke remain green

### Out Of Scope
- brand redesign
- board/map redesign

## Issue 5

### Title
Freeze the `v2.3` consolidation milestone and record follow-on boundaries

### Objective
- document `v2.3` as a concise-shell and game-history foundation milestone, not a replay or agent milestone

### Acceptance Criteria
- docs describe the saved-history foundation accurately
- docs keep replay/highlights in later follow-on work
- docs place bot-skill generation from recorded games in `v2.4`, not `v2.3`
- docs keep `v2.4` UX and `v3` agent work clearly separate

### Validation
- docs review against shipped `v2.3` behavior

### Out Of Scope
- launching replay/highlight features
- expanding into `human+agent`

## Notes
- The key product discipline for `v2.3` is restraint.
- `v2.3` should make the current product cleaner and more instrumented, not more conceptually crowded.
