## Recommended Execution Order
1. persist a canonical multiplayer game history log
2. prove that saved history can be fetched and reconstructed reliably
3. redesign landing with a transit-atmospheric hero and two clear mode choices
4. refactor online setup into a gateway plus dedicated child flows
5. redesign the start-game and join-room setup pages to match the new shell
6. harden setup shell navigation, reconnect placement, and media fallback behavior
7. clean up typography/color drift and align the shipped surface with the current design language
8. freeze docs and record what stays for `v2.4` and `v3`

## Issue 1

### Title
Persist a canonical ordered game history for every multiplayer match

### Objective
- ensure the authoritative backend stores enough ordered move data to reconstruct a full match after it ends

### Acceptance Criteria
- every completed multiplayer game has a durable ordered history
- history uses `action log + checkpoint snapshots`
- history records include acting seat, move type, turn/round sequencing, and endgame outcome anchors
- the stored data is clear enough to review a game move-by-move like a chess record
- history capture does not change game rules or authoritative validation behavior

### Validation
- deterministic server tests for complete-game history persistence

### Out Of Scope
- replay UI
- AI-generated summaries
- local pass-and-play history

## Issue 2

### Title
Expose a minimal retrievable match-history surface for review and future replay consumers

### Objective
- make the saved history usable instead of only stored

### Acceptance Criteria
- a game history can be fetched in order after game completion
- the returned data is sufficient for internal review of each move and round progression
- the contract is stable enough for future replay/highlight tooling
- the first retrieval surface stays internal or low-prominence

### Validation
- end-to-end server test for history fetch after a completed match

### Out Of Scope
- polished user-facing replay experience

## Issue 3

### Title
Redesign landing into a transit-atmospheric hero with only two mode choices

### Objective
- make Hudson Hustle open with one strong branded decision instead of a setup-heavy first screen

### Acceptance Criteria
- landing uses a full-screen transit-atmospheric background with a static shipped surface and optional video enhancement
- landing keeps only:
  - the Hudson Hustle title
  - `Local`
  - `Online`
- the first screen removes explanatory product copy and stacked setup panels
- mode choices read as one cohesive glass/frosted family instead of heavy cards

### Validation
- browser smoke on landing interactions and fallback behavior
- one findings-first visual review

### Out Of Scope
- online form redesign beyond the mode-choice layer

## Issue 4

### Title
Refactor online setup into a gateway with `Start game` and `Join room`

### Objective
- turn online entry into a second-layer decision screen instead of a full setup dashboard

### Acceptance Criteria
- online entry becomes a gateway screen with:
  - a large `Online` title
  - a short subtitle
  - `Start game`
  - `Join room`
  - a visible `Back`
- reconnect is removed as a full competing setup panel from the gateway
- the gateway and landing share one visual family

### Validation
- browser smoke on entering and leaving online mode
- one findings-first visual review

### Out Of Scope
- detailed host/join form redesign

## Issue 5

### Title
Redesign the dedicated `Start game` and `Join room` child flows

### Objective
- move real setup input into separate child pages that preserve the new concise shell language

### Acceptance Criteria
- `Start game` becomes a dedicated host wizard with:
  - a left rail
  - one primary control surface
  - a reduced step sequence
- `Join room` becomes a dedicated join wizard with the same shell language
- both pages preserve the same setup shell language and keep visible `Back`
- reconnect is folded into `Join room` as a secondary recovery path
- setup no longer reads like several stacked dashboard panels

### Validation
- browser smoke for:
  - create room
  - preview room
  - join room
  - reconnect path

### Out Of Scope
- live gameplay shell redesign beyond setup pages

## Issue 6

### Title
Harden setup-shell navigation, recovery placement, and media fallback behavior

### Objective
- keep the new setup flow coherent and reliable instead of visually cleaner but brittle

### Acceptance Criteria
- every setup layer keeps a visible `Back` action where applicable
- transit media behavior supports both optional autoplay video and a graceful static-first fallback
- reconnect remains available inside `Join room` without competing with the primary flow
- mobile and narrow-width behavior remain usable
- room lobby aligns with the same atmospheric shell and control-surface language instead of reverting to dashboard styling

### Validation
- browser smoke on:
  - back navigation
  - video fallback
  - reconnect field visibility and submission

### Out Of Scope
- hosted media pipeline optimization beyond what is needed for a stable shipped fallback

## Issue 7

### Title
Align gameplay typography, colors, and layout tokens with the current design direction

### Objective
- remove lingering legacy style drift so the product feels visually coherent

### Acceptance Criteria
- legacy font/color usage in shipped gameplay surfaces is replaced or normalized
- layout spacing and hierarchy feel tighter and more intentional
- style cleanup does not silently change gameplay behavior

### Validation
- visual review against current gameplay surfaces
- web build and browser smoke remain green

### Out Of Scope
- brand redesign
- board/map redesign

## Issue 8

### Title
Freeze the `v2.3` consolidation milestone and record follow-on boundaries

### Objective
- document `v2.3` as a concise-shell and game-history foundation milestone, not a replay or agent milestone

### Acceptance Criteria
- docs describe the saved-history foundation accurately
- docs keep replay/highlights in later follow-on work
- docs place bot-skill generation from recorded games in `v2.4`, not `v2.3`
- docs keep `v2.4` UX and `v3` agent work clearly separate

### Validation
- docs review against shipped `v2.3` behavior

### Out Of Scope
- launching replay/highlight features
- expanding into `human+agent`

## Notes
- The key product discipline for `v2.3` is restraint.
- `v2.3` should make the current product cleaner and more instrumented, not more conceptually crowded.
