# V2.1 Shell Review Memo

## Purpose

This memo consolidates the latest system and layout reviews across the active `v2.1` shell work.

It is not the canonical design-system spec.
It records the most actionable findings from the current review pass so the repo can turn them into concrete UI changes.

## Inputs

This memo synthesizes four review lanes:
- system consistency review
- layout critique
- active game shell layout options
- Duolingo system-behavior reference study

## What Is Working

The strongest system anchors remain:
- `StatusBanner`
- `StateSurface`
- `ModalShell`

These work because each one has:
- one dominant message
- one supporting layer
- one clear role

The system is now much closer to a coherent family than earlier `v2.1` slices.
The remaining problems are mostly about layout weight and role boundaries, not about color direction or typography choice.

## Main Findings

### 1. Layout sameness is the biggest remaining weakness

Too many surfaces still read as:
- header
- body
- extras

with roughly equal weight.

That makes them feel more templated than authored.

### 2. `SectionHeader` needs stricter density rules

The same eyebrow/title/meta rhythm appears too often.

That is useful for consistency, but it flattens hierarchy.

`SectionHeader` should now be treated as three densities:
- `ceremony`
- `standard`
- `compact`

Recommended use:
- `ceremony`
  - `Round table`
  - modal / tutorial main titles
  - special reveal moments
- `standard`
  - board and action shell sections
  - normal setup / lobby sections
- `compact`
  - inventory / supply / supporting sections such as hand, tickets, and market

### 3. `Panel` and `SurfaceCard` still need a harder boundary

Target distinction:
- `Panel` = structural shell frame
- `SurfaceCard` = nested decision / detail / summary object

When `SurfaceCard` feels like a smaller `Panel`, the shell loses clarity.

### 4. `ChoiceChipButton` must read as a decision shelf, not appended buttons

The style has improved, but composition matters as much as the button paint.

Route and city detail should separate:
- information zone
- decision zone

The action shelf should feel like a dedicated payment tray or build tray, not a loose row of controls after text.

### 5. Tutorial composition is still the weakest layout in the system

The current guide was moving in the right direction, but still felt too much like app onboarding.

The better direction is:
- left rail = guide spine
- right side = one main lesson block plus one supporting block

Not three equal detail cards.

## Active Game Layout Directions

### Recommended for `v2.1`

`Weighted Triptych`

Interpretation:
- board remains central and dominant
- left side becomes a more stable persistent strip
- right / lower decision area becomes more explicitly about the current move

Practical implications:
- `Round table` stays most ceremonial
- private info becomes a grouped working-memory stack
- market feels lighter and more supporting
- action rail behaves more like a decision dock

### Worth exploring later

`Board Stage + Docked Rail`

This is a stronger game-shell direction, but more disruptive than the current `v2.1` tolerance.

### Experimental only

`Atlas Desk`

Useful as a future experiment, not a `v2.1` target.

## Duolingo Translation

Duolingo is useful here as a system-behavior reference, not as a visual reference.

### Adopt
- guided next action
- tiered hierarchy
- strong consistency discipline
- milestone feedback stronger than routine feedback

### Adapt
- guided flow into:
  - setup progression
  - reconnect clarity
  - selected object -> available action logic
  - tutorial sequencing

### Avoid
- mascot / brand energy
- habit-gamification loops
- over-soft rounded module sprawl
- chirpy encouragement tone

## Actionable Priorities

### Done in this pass
- add explicit `SectionHeader` density rules in code
- move key modal / tutorial titles into ceremony use
- regroup private info and supply hierarchy in the game shell
- turn route / city detail actions into a more explicit decision shelf
- simplify tutorial composition toward lesson-sheet structure

### Still worth doing next
- reduce header repetition in setup / lobby where sections are still too equal
- tighten `Panel` vs `SurfaceCard` visual boundary further
- continue shifting the active shell toward:
  - persistent strip
  - decision dock
  while keeping the map first

## Boundary

This memo supports `v2.1`.

It does not authorize:
- map redesign
- board label overhaul
- a full shell re-architecture
- Duolingo-like brand treatment
