# V2.3 Concise UI And Game History Plan

## Summary
- This plan implements [V2.3 Concise UI And Game History PRD](/Users/djfan/Workspace/HudsonHustle/docs/product/v2/v2.3-concise-ui-and-game-history-prd.md).
- Milestone target: make the shipped `v2` experience more concise and durable before a larger `v2.4` UX push.
- Product goal:
  - simplify and modernize the current live UI without reopening core gameplay or a broad UX redesign
  - persist a complete per-game move history so future replay, personalized highlight generation, and bot-skill learning have a trustworthy foundation

## Scope Boundary

### In Scope
- remove or simplify live UI elements that add noise without carrying core gameplay value
- tighten gameplay shell layout so the main play surface reads faster and more cleanly
- replace lingering legacy typography/color/style drift with the current system direction
- define a canonical persisted game-history model for multiplayer games
- store enough ordered data to reconstruct a game's full move/round flow after completion
- expose a minimal internal/server-side retrieval surface for saved game history
- use `action log + checkpoint snapshots` as the first shipped history shape
- keep the history format useful for future:
  - replay generation
  - player-facing highlights
  - strategy review
  - later `v2.4` bot-skill learning and heuristic review

### Out Of Scope
- full replay UI in the product
- AI-written post-game highlight generation
- automated strategy coaching in the shipped app
- bot self-improvement or online learning loops
- bot-skill generation from recorded games before `v2.4`
- broader `v2.4` UX redesign work
- `v3` controller work such as `human+agent` or full autonomous `agent` seats
- gameplay rule, scoring, or balance changes unless required to correctly log history
- local pass-and-play history capture

## Slice Order

### Slice 1
- Objective:
  - define and persist the canonical multiplayer game-history log for authoritative actions and phase transitions using `action log + checkpoint snapshots`
- Why first:
  - this is the riskiest new system in `v2.3`
  - if the log shape is wrong, later replay, review, and training work will inherit bad data
- Likely areas touched:
  - `apps/server/src/room-service.ts`
  - server persistence/storage modules
  - `packages/game-core` action/result typing only if shared event metadata is required
  - database schema/migration files
- Validation:
  - deterministic server tests proving that a finished game records a complete ordered timeline
  - verify actor seat, action type, timing/turn index, and resulting checkpoint references are stable
  - verify the log is detailed enough to reconstruct the game move-by-move in a chess-like review flow
- Guardrail:
  - do not build player-facing replay UI yet
  - do not turn the whole backend into a fully generalized event-sourced system

### Slice 2
- Objective:
  - make saved game history retrievable and reconstructable for internal review and future product surfaces
- Why here:
  - once the log exists, the next value is proving it can actually drive downstream consumers
- Likely areas touched:
  - server read endpoints or internal fetch helpers
  - persistence query layer
  - lightweight serialization helpers for timeline export/review
  - docs describing the history contract
- Validation:
  - one end-to-end test that creates a game, finishes it, and fetches a usable ordered history
  - verify a consumer can distinguish:
    - round/turn boundaries
    - acting seat
    - move type
    - post-move public state anchors
    - final outcome
- Guardrail:
  - keep the first retrieval surface internal or low-prominence for online multiplayer only
  - no polished replay viewer yet

### Slice 3
- Objective:
  - simplify the live UI shell and remove legacy style drift so the shipped `v2` interface feels intentional and concise
- Why here:
  - the logging foundation is the technical risk; UI simplification is important but should not destabilize persistence work
- Likely areas touched:
  - `apps/web/src/App.tsx`
  - gameplay shell and panel components under `apps/web/src/components/`
  - `apps/web/src/styles.css`
  - typography/color tokens if the current design system needs cleanup
  - showcase/docs only where shipped behavior changes
- Validation:
  - browser smoke for setup, lobby, in-game turn flow, and endgame
  - visual review that confirms reduced chrome and improved hierarchy without hiding needed game state
- Guardrail:
  - keep this reductive, not expansive
  - no new large feature surfaces or major information architecture redesign

### Slice 4
- Objective:
  - freeze `v2.3` docs and record the bridge into `v2.4` and `v3`
- Why here:
  - `v2.3` should clearly read as a consolidation milestone, not the start of a sprawling new product branch
- Likely areas touched:
  - `docs/product/prd.md`
  - `docs/product/tech-spec.md`
  - `docs/product/v2/v2-mvp-architecture.md`
  - player-facing docs only if visible gameplay shell wording changes
- Validation:
  - findings-first review of the shipped `v2.3` delta
  - doc check that replay/highlights/agent work are still clearly future scope
- Guardrail:
  - do not silently smuggle `v2.4` UX work into the freeze pass

## Risks
- logging too little detail and discovering later that replay or analysis cannot reconstruct important choices
- logging too much raw state and creating an unnecessarily heavy persistence model
- UI simplification accidentally hiding necessary game-state context for new players
- style cleanup drifting into subjective redesign churn instead of concrete readability wins
- history data becoming server-only in a way that is hard to reuse for future review/highlight pipelines

## Chosen Decisions
- `v2.3` history scope is online multiplayer only
- the first shipped history model is `action log + checkpoint snapshots`
- the required fidelity is enough to review a match move-by-move and round-by-round, similar to a chess game record
- replay UI is deferred
- game-derived bot-skill generation is deferred to `v2.4`
- the first retrieval surface should stay internal or otherwise low-prominence
