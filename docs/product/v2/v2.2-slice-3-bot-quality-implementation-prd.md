# V2.2 Slice 3 Implementation PRD

## Product Summary

This document scopes the third implementation slice of `v2.2` system player work.

Slice 3 improves the first public `bot` seat from “legal and minimally credible” into a competent deterministic baseline that can finish games in a way that feels understandable and not obviously self-destructive.

## Problem

Slices 1 and 2 proved the architecture and exposed `bot` seats in normal room setup, but the current policy is still deliberately shallow:
- it can complete turns legally
- it can make simple ticket-aligned claims
- it can make deterministic draw choices

That is enough for architecture proof and setup flow, but not enough for a real `v2.2` milestone. If the bot remains too weak or too naive:
- mixed-room play will feel like a system demo rather than a credible feature
- later difficulty or `human+agent` work will be shaped around a poor baseline
- the product will ship a public `bot` that is legal but obviously unsatisfying

## Goals

- Improve the built-in `bot` to a competent baseline without changing its deterministic/server-owned nature.
- Reuse current Hudson Hustle strategy heuristics where they clearly improve move quality.
- Keep the policy understandable and testable through fixed-state decisions.
- Preserve hidden-information boundaries and the existing authoritative action path.

## Non-Goals

- No difficulty tiers yet.
- No stochastic personality layer or human-like bluffing behavior.
- No explanation UI or natural-language coaching.
- No external model-driven planning.
- No gameplay rules or balance changes in this slice.

## User Stories

1. As a player, I want a `bot` to make moves that look credible enough to finish a game without obviously sabotaging itself.
2. As a maintainer, I want the stronger baseline to stay deterministic enough for regression testing and debugging.
3. As a future designer of difficulty tiers, I want one clearly defined competent baseline before introducing easier or harder variants.

## Solution

Slice 3 should improve the bot policy in bounded, testable layers:
- stronger starting-ticket keep logic
- better claim prioritization tied to ticket progress and blocking value
- better draw behavior based on affordable claim plans rather than purely fallback determinism
- safer handling of turn patterns where the bot might otherwise waste a turn or choose an obviously poor route

The implementation should remain:
- server-side
- deterministic or seed-controlled
- built on the same public/private seat view
- routed through the same authoritative action path

The product shape should be:
- one stronger baseline `bot`

It should not become:
- a difficulty-selection system
- a general planning engine with public tuning controls

## Scope

### In Scope

- improve bot heuristics for:
  - initial ticket keep choices
  - claim prioritization
  - draw-card decisions
  - avoiding obvious self-destructive turns
- add read-only evaluation helpers if they materially simplify deterministic policy quality
- fixed-state tests that prove stronger decision quality in representative scenarios
- docs updates if strategy or terminology references need alignment

### Out Of Scope

- difficulty levels
- randomized personality variants
- player-facing explanation or hint surfaces
- timer or reconnect hardening
- any rule or scoring changes

## Implementation Decisions

- The baseline bot remains server-owned and deterministic.
- The decision layer may add read-only evaluation helpers in shared code if that improves quality and keeps policy logic clear.
- Heuristics should favor:
  - advancing owned ticket plans
  - taking strong low-risk claims when affordable
  - drawing toward a credible near-term claim rather than random accumulation
- The bot should still avoid seeing any information beyond the seat-appropriate public/private projection.
- This slice should prefer a small number of strong heuristics over a sprawling pseudo-AI policy.

## Testing Decisions

- Add deterministic tests for representative board states where the bot should:
  - keep the better starting-ticket subset
  - choose a ticket-aligned claim over a weaker generic claim
  - draw toward a plausible claim plan when no good claim exists
  - avoid obviously bad or illegal choices
- Preserve existing Slice 1 and Slice 2 regression coverage.
- Prefer behavior-first expectations over testing internal scoring math directly unless the helper itself is shared and worth unit-testing.

## Open Questions

- Which route-evaluation heuristics are mandatory for the first competent baseline versus better left for a later difficulty-tier slice?
- Should blocking value against visible opponents be part of the baseline, or deferred until after ticket/claim quality is stronger?
- Is there one read-only helper that should move into shared code now to avoid duplicating route-progress reasoning inside the server policy?
