# V2.3 Slice 3 Setup Redesign Note

## Summary
- `v2.3` should redesign the landing and online setup flow into a layered, low-copy, transit-atmospheric entry sequence.
- The goal is not to add more setup UI. The goal is to remove hesitation by making each screen ask for one decision only.
- This redesign is intentionally limited to:
  - landing
  - online gateway
  - start-game flow
  - join-room flow
  - room lobby shell
  - setup shell navigation and media behavior

## Experience Goal
- a new player opens Hudson Hustle and sees one strong branded scene
- the first decision is only:
  - `Local`
  - `Online`
- if the player chooses `Online`, the next decision is only:
  - `Start game`
  - `Join room`
- form fields appear only after the player chooses one of those two branches
- every setup screen keeps a visible top-right `Back` action

## Visual Direction
- full-screen transit atmosphere on landing, with a static background image as the default shipped surface and video treated as optional enhancement
- the background should feel like NYC / NJ platform space, not a generic city loop
- a darkened atmospheric overlay should protect contrast without hiding the transit image entirely
- the interactive layer should read like smoked glass / station control surfaces, not like heavy dashboard cards
- `Fraunces` stays for the page-level hero title
- `Inter` stays for controls, labels, and setup metadata

## Information Architecture

### Screen 1
- `Hudson Hustle`
- `Local`
- `Online`

Rules:
- no explanatory paragraph
- no feature list
- no extra panels

### Screen 2
- title: `Online`
- short subtitle
- `Start game`
- `Join room`
- top-right `Back`

Rules:
- this is a gateway, not a form page
- reconnect does not live here as a full third option

### Screen 3A
- title: `Start game`
- host setup wizard with a left rail and one primary control card
- visible `Back`

Step order:
- host
- seats
- board

### Screen 3B
- title: `Join room`
- join wizard with the same rail + control-card structure
- visible `Back`

Step order:
- room code
- seat
- enter room

Recovery:
- reconnect token stays inside the join flow as a secondary path

### Lobby
- the room lobby should use the same atmospheric shell language
- the lobby should read like a boarding / departure state, not an admin dashboard
- the lobby keeps:
  - a rail for room identity and session state
  - a main control surface for seats and readiness
  - a bottom action bar for leave / ready / start

## Design Rules
- each screen gets one title and one primary action zone
- copy stays short and functional
- no repeated explanatory surfaces above forms
- reconnect should feel like a recovery path, not a third competing primary mode
- setup and lobby should feel like one family:
  - atmospheric shell
  - rail-guided flow
  - single control surface
- setup pages may redesign information architecture
- gameplay surfaces should not be redesigned here beyond what `Issue 3` already covers

## Media And Fallback Rules
- if video is used, it should autoplay muted, loop, and play inline
- the shipped setup shell must work with a static atmospheric background alone
- the fallback should preserve readability and the same composition
- implementation should avoid a brittle dependency on one hosted media source

## Output Shape
- landing becomes a transit-atmospheric hero with two glass mode tiles
- online becomes a two-choice gateway
- start-game and join-room become separate wizard flows with shared shell language
- lobby aligns with the same shell and control-panel family
- setup flow reads as one cohesive family instead of several stacked dashboard panels
