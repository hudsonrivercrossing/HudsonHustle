## Issue 1

### Title
Add mixed human/bot seat intent to normal multiplayer room setup state

### Objective
- Extend room setup state and request handling so normal multiplayer setup can represent human and `bot` seats up to room size.

### Acceptance Criteria
- Normal setup state can represent any mix of human and `bot` seats up to room size.
- Existing human-only setup still works without extra required input.
- Server room-create handling accepts and persists the mixed seat intent.

### Validation
- targeted server tests for room creation/setup payload handling
- regression coverage for human-only room creation

### Out Of Scope
- setup visual polish beyond the minimum needed
- bot strategy changes

## Issue 2

### Title
Expose bot seat selection in MultiplayerSetupScreen without heavy controller UI

### Objective
- Add the minimum seat-oriented setup controls so a host can choose one or more `bot` seats in the normal room setup flow.

### Acceptance Criteria
- Hosts can assign one or more seats as `bot` seats during setup.
- The setup UI remains readable and clearly lighter than an admin/control panel.
- Human-only setup remains fast and visually close to `v2.1`.

### Validation
- browser verification for one-bot and multi-bot room setup
- design review against existing shell/system hierarchy

### Out Of Scope
- difficulty settings
- advanced controller management

## Issue 3

### Title
Render mixed human/bot seats clearly in lobby and start flow

### Objective
- Make mixed rooms understandable once created by showing `bot` seats clearly in lobby/start surfaces.

### Acceptance Criteria
- Lobby seat rows clearly distinguish `bot` and human seats.
- Mixed rooms can proceed through the normal start flow.
- Human seats keep existing join/ready semantics while bot seats read as server-owned/non-reconnectable.

### Validation
- browser tests for mixed-room lobby and start progression
- targeted server tests for seat representation at room start

### Out Of Scope
- reconnect hardening beyond current semantics
- broad lobby redesign

## Issue 4

### Title
Prove normal mixed-room setup and start regression safety

### Objective
- Ensure the first public bot setup path does not regress stable `v2.1` multiplayer behavior.

### Acceptance Criteria
- One-bot and multi-bot rooms can reach game start correctly.
- Human-only rooms still follow the existing setup/start path.
- Mixed rooms do not bypass the normal authoritative lifecycle.

### Validation
- browser coverage for:
  - one bot
  - multiple bots
  - human-only regression
- targeted server coverage for setup/start lifecycle

### Out Of Scope
- timer hardening
- reconnect edge-case expansion
