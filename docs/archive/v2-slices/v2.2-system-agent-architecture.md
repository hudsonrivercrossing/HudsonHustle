# V2.2 System Agent Architecture Recommendation

## Summary

The first system player in Hudson Hustle should be a **built-in server-side deterministic bot** that operates through the existing authoritative backend and shared `game-core` rules.

This is the safest, cheapest, and most extensible first step.

## Recommended First Path

### Phase 1
- Add internal controller type: `bot`
- Keep it server-owned
- Feed it only the same public/private view a real seat should see
- Make it submit normal legal action intents through the authoritative server path

Current Slice 1 proof now covers:
- server-owned bot seats with no reconnect secret
- authoritative room-seat model that distinguishes client-owned and server-owned controllers
- deterministic baseline policy for:
  - initial ticket confirmation
  - simple ticket-aligned route claims
  - deterministic card draw choices when no obvious claim is available

### Phase 2
- Expose `bot` seat selection through normal room setup
- Support mixed human/bot rooms up to room size
- Keep lobby/start semantics understandable for reconnectable humans and server-owned bot seats

Current Slice 2 result now covers:
- normal multiplayer setup can prefill one or more non-host seats as `bot`
- mixed rooms can proceed through the normal lobby and start flow
- human-only rooms stay behaviorally unchanged

### Phase 3
- Improve heuristics and planning depth
- Add difficulty levels through evaluation depth or strategy variation
- Keep decision-making deterministic or seed-controlled where possible

Current Slice 3 result now covers:
- coherent starting-ticket selection
- route-demand-based claim prioritization
- deterministic draw choices aimed at stronger near-term claims
- one competent baseline policy, still without difficulty tiers

Current Slice 4 result now covers:
- timed human turns can hand off through server-owned bot turns without stalling
- restored mixed rooms resume server-owned turns immediately when a bot seat is active
- reconnect semantics remain explicitly human-only while mixed rooms stay operationally stable

Current Slice 5 freeze confirms:
- the first public `v2.2` system-player milestone is public mixed human/`bot` play, not internal-only bot infrastructure
- `bot`, `agent`, and `human+agent` remain distinct concepts
- difficulty tiers and broader agent work remain follow-on work, not part of the frozen milestone

### Phase 4
- Add `human+agent`
- Use agent suggestions and explanation
- Keep final move confirmation human-owned

### Phase 5
- Explore autonomous external agents only after the internal path is stable

## Option Comparison

### Built-in server bot
Best first choice.

Pros:
- authoritative by default
- easy to test
- easy to replay
- low hosting complexity
- lower hidden-information risk

Cons:
- less expressive at first
- requires deliberate heuristic design

### Deterministic decision adapter over `game-core`
This is the recommended implementation shape for the first bot.

Pros:
- aligned with current architecture
- bounded and testable
- easy to debug against fixed seeds

### External model-driven agent
Not recommended as a first milestone.

Pros:
- richer long-term experimentation

Cons:
- nondeterministic
- harder to debug
- higher latency and cost
- weaker operational reliability
- more hidden-information and fairness concerns

### User-supplied CLI / external autonomous agents
Definitely not first.

Interesting later, but wrong for the first system-player milestone.

## Major Risks
- hidden information leakage
- nondeterministic behavior breaking replay/debugging
- timer and latency interaction
- unfair or frustrating play quality
- confusion between `bot`, `agent`, and `human+agent`

## Non-Goals For First Iteration
- no external LLM dependency
- no user-provided autonomous agents
- no marketplace/plugin protocol
- no chat-first “assistant” system before controller abstractions are stable

## What Should Not Be First
- external LLM autonomous seat
- user-provided CLI agent protocol
- `human+agent` chat before a stable internal bot/controller path
- any route that bypasses the authoritative server
