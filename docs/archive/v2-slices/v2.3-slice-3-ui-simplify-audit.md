# V2.3 Slice 3 UI Simplify Audit

## Purpose

This document narrows `Issue 3` into a short simplify target list before implementation starts.

It is not a new roadmap issue and not a broad redesign brief.
It exists to keep `v2.3` reductive:
- remove low-value chrome
- merge duplicated information
- reduce visual weight where the information is still needed

## Design Rule

`v2.3` should make the current multiplayer shell feel more concise, not more feature-rich.

The working rule for every surface is:
- if it does not help a player decide, orient, or recover, cut it
- if it repeats nearby information, merge it
- if it is useful but too loud, keep it and lower its weight

## Simplify Target List

### 1. Setup Entry And Multiplayer Setup

#### Delete
- extra explanatory copy that repeats the mode distinction after the first decision
- any setup chrome that makes create/join feel like separate mini-products instead of one room-entry flow

#### Merge
- `Create room` controls into a tighter single host form
- `Join room` preview and seat-pick flow into one clearer guest path
- timer and seat-plan explanations into the form labels/helpers instead of separate visual emphasis blocks

#### De-emphasize
- release/config metadata during setup
- ceremonial headers in setup panels

### 2. Lobby

#### Delete
- repeated lobby explanation copy once the room code and seat table are already visible
- any non-actionable timer placeholder text before the match starts

#### Merge
- top status banner and session details so room state is not explained twice
- seat readiness and occupancy understanding into one cleaner table scan

#### De-emphasize
- config/version details
- reconnect artifact weight
- secondary seat-state chips where text already communicates the same thing

### 3. In-Game Topbar

#### Delete
- non-essential chrome competing with turn status
- always-visible helper affordances that are not needed every turn

#### Merge
- map/config metadata into one quieter session marker
- leave/reconnect utilities into a lighter utility cluster

#### De-emphasize
- `ScoreGuide`
- config summary tooltip behavior
- topbar action emphasis outside the current-turn status

### 4. Left Rail

#### Delete
- stats that do not materially help turn-by-turn decisions
- decorative separation that makes one sidebar feel like four unrelated products

#### Merge
- `Your hand` and supply interaction into one more coherent card-economy zone if possible
- scoreboard facts into a simpler scan-first player list

#### De-emphasize
- ticket progress ornamentation
- secondary player stats that are useful but not primary

### 5. Action Rail

#### Delete
- verbose empty-state teaching copy
- disabled pseudo-buttons used only as labels, especially timer-as-button treatment

#### Merge
- board inspection state and action choice into one tighter decision surface
- route and city detail copy so the action area reads like a decision sheet, not a tutorial card

#### De-emphasize
- instructional prose once a route or city is selected
- extra labels that duplicate what the board object already shows

### 6. Endgame

#### Delete
- any remaining ceremony that slows result scanning without adding clarity

#### Merge
- score hero and breakdown hierarchy into a more table-like result surface

#### De-emphasize
- repeated card-like framing if it overstates each result block

### 7. Legacy Style Drift

#### Delete
- old visual treatments that survive only by inheritance

#### Merge
- typography hierarchy across setup, lobby, and in-game shell so they read as one shipped system
- chip/pill/panel usage so state markers and chrome markers do not blur together

#### De-emphasize
- over-strong shadows, eyebrow density, and mixed title weights that make every section compete equally

## Not In Scope For Slice 3

- no replay UI
- no new product surfaces
- no gameplay rule changes
- no board/map redesign
- no broad `v2.4` UX rethink

## Execution Guidance

Implement `Issue 3` in this order:

1. topbar + lobby redundancy reduction
2. in-game left rail and action rail simplification
3. setup flow tightening
4. endgame polish pass
5. token/style drift cleanup that the previous steps prove necessary

## Success Criteria

`Issue 3` is successful when:
- setup, lobby, gameplay, and endgame all read faster
- the board remains the focal surface
- private information stays clear
- the shell contains less repeated explanation
- no new feature surfaces are introduced just to replace removed noise
