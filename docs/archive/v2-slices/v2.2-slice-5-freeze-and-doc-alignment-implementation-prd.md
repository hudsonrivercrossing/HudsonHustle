# V2.2 Slice 5 Implementation PRD

## Product Summary

This document scopes the fifth and final implementation slice of `v2.2` system player work.

Slice 5 does not add new controller behavior. It freezes the first public `v2.2` system-player milestone, aligns terminology across product and engineering docs, and captures the remaining follow-on boundary without reopening scope.

## Current Outcome

Slice 5 is now implemented:
- one findings-first freeze review concludes there are no blocker findings
- core `v2` architecture and tech docs now describe the same shipped `bot` / `controller` model
- multiplayer and player-facing wording now reflect mixed human/`bot` rooms without implying `bot` reconnect behavior
- the first public `v2.2` system-player milestone is explicitly frozen and follow-on work is clearly separated

## Problem

Slices 1 through 4 now ship a real public mixed human/`bot` room path:
- server-owned `bot` seats exist in the authoritative room model
- normal multiplayer setup can assign one or more `bot` seats
- the baseline `bot` is deterministic and competent enough to finish games credibly
- mixed-room timer, reconnect, and persistence behavior is hardened

At this point the biggest risk is no longer implementation drift. It is documentation and terminology drift:
- different docs can still describe `bot`, `agent`, `human+agent`, `seat`, and `controller` with different implied meanings
- some docs still read like `bot` is internal-only while others describe it as publicly available
- follow-on boundaries can become muddy if Slice 5 does not explicitly freeze what `v2.2` is and is not

That would make the milestone harder to operate, harder to extend, and easier to accidentally reopen.

## Goals

- Run one findings-first freeze review pass against the shipped `v2.2` system-player milestone.
- Align the core `v2` docs so they describe the same shipped controller model and same shipped mixed-room behavior.
- Keep the distinction between:
  - `bot`
  - `agent`
  - `human+agent`
  - `seat`
  - `controller`
  explicit and stable.
- Freeze the milestone by documenting what is done and what belongs to a later slice or later version.

## Non-Goals

- No new bot-strength heuristics.
- No difficulty tiers.
- No external agents or `human+agent` implementation.
- No UI redesign or design-system side work.
- No new multiplayer runtime behavior beyond small clarifying doc-backed wording changes.

## User Stories

1. As a maintainer, I want one stable written account of what the first `v2.2` system-player milestone actually ships.
2. As a future planner, I want the docs to make later work legible without pretending it is already in scope.
3. As a reviewer, I want a short findings-first freeze review that records residual risk instead of leaving the milestone implicitly “done.”

## Solution

Slice 5 should close the first `v2.2` milestone in four steps:
- capture one findings-first freeze review pass with only residual issues or explicit “no blocker” conclusions
- align the core `v2` architecture and tech docs to the same shipped controller/runtime terminology
- align the player-facing and multiplayer-flow docs anywhere mixed-room `bot` behavior is now user-visible
- freeze the roadmap and `v2` doc index so the next work starts from an explicit post-Slice-5 boundary instead of reopening the same questions

## Scope

### In Scope

- one findings-first freeze review document for the shipped `v2.2` system-player milestone
- terminology alignment in:
  - `docs/product/tech-spec.md`
  - `docs/product/v2/v2-mvp-architecture.md`
  - `docs/product/v2/v2.2-system-agent-architecture.md`
  - `docs/product/v2/v2-multiplayer-flow.md`
  - `docs/product/v2/README.md`
- player-facing wording updates only where shipped mixed-room behavior is now relevant
- explicit follow-on boundary for post-`v2.2` controller work

### Out Of Scope

- implementation of new controller types
- design-system or setup-flow redesign
- speculative `v2.3` product planning beyond a short boundary statement
- release automation or deployment changes

## Implementation Decisions

- Freeze review should be findings-first, ordered by severity, and explicitly allowed to conclude “no blocker.”
- Terminology should use:
  - `seat` for the stable position in a room
  - `controller` for the thing that owns decision-making for that seat
  - `bot` for the first shipped server-owned autonomous controller
  - `agent` for broader future non-human controller concepts
  - `human+agent` for future assisted play, not current `v2.2` behavior
- Docs should state plainly that the shipped `v2.2` milestone includes public mixed human/`bot` rooms.
- Slice 5 should end with a clear “milestone frozen” statement and a narrow statement of what comes next.

## Testing Decisions

- No new gameplay behavior should require new automated test coverage in this slice unless a doc alignment change reveals a real mismatch.
- Validation for Slice 5 is primarily:
  - findings-first review pass
  - terminology consistency review across affected docs
  - sanity check that no code drift was introduced while documenting

## Follow-On Questions

- Should post-`v2.2` work add difficulty tiers before any `human+agent` work, or keep focus on assistance before stronger autonomy?
- Does the first public `bot` milestone justify a dedicated player-facing explainer later, or is multiplayer setup/lobby copy sufficient?
