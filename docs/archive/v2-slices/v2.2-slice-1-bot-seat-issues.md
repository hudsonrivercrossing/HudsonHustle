## Issue 1

### Title
Introduce internal bot seat/controller state in the authoritative room model

### Objective
- Add the minimum room/seat/controller modeling needed for one server-owned `bot` seat path.

### Acceptance Criteria
- Room/seat data can represent a bot-controlled seat without breaking human seats.
- Human-only room behavior remains unchanged.
- The representation is seat-oriented and does not special-case “magic bot room state” outside the normal model.

### Validation
- targeted server tests for room/seat/controller representation
- regression coverage for existing human room lifecycle

### Out Of Scope
- normal setup UX for choosing bot seats
- bot action quality

## Issue 2

### Title
Add a server-owned legal action path for bot turns

### Objective
- Let the server detect when the active seat is a bot seat and submit a legal action through the normal authoritative path.

### Acceptance Criteria
- A bot-controlled active seat can complete a legal turn without client ownership.
- The bot does not mutate game state directly outside the normal room/game action path.
- Turn progression remains valid after bot actions.

### Validation
- deterministic server tests proving:
  - bot acts only on its own active turn
  - submitted actions are legal
  - turn advancement stays correct

### Out Of Scope
- sophisticated strategy
- multiple bot-seat UX

## Issue 3

### Title
Implement a deterministic baseline bot policy for Slice 1

### Objective
- Provide the smallest deterministic policy that allows the bot seat to participate credibly in turn flow.

### Acceptance Criteria
- The baseline policy is deterministic or seed-controlled.
- It can choose legal actions in at least the initial supported scenarios.
- It is not obviously self-destructive in the first supported turn patterns.

### Validation
- fixed-state tests against known bot decisions
- regression tests for supported turn types

### Out Of Scope
- difficulty tiers
- broad strategic strength tuning
- explanation UI

## Issue 4

### Title
Prove hidden-information and human-room regression safety for the first bot path

### Objective
- Make sure the new bot path does not leak more information than a seat should see and does not regress existing human-only rooms.

### Acceptance Criteria
- The bot observes only seat-appropriate public/private state.
- Human-only rooms still start, play, reconnect, and finish exactly as before.
- The first bot path does not weaken room auth or hidden-information boundaries.

### Validation
- deterministic server tests for projection/isolation behavior
- existing multiplayer regression suite remains green

### Out Of Scope
- full mixed human/bot setup UX
- external agents

## Issue 5

### Title
Document the internal Slice 1 bot-seat path and update v2 architecture docs

### Objective
- Keep the architectural proof aligned with repo docs before Slice 2 exposes bot seats in normal setup.

### Acceptance Criteria
- `v2.2` docs reflect that Slice 1 is an internal architecture proof.
- terminology stays consistent:
  - `bot`
  - `seat`
  - `controller`
- follow-on boundaries for Slice 2 remain explicit.

### Validation
- docs review against shipped Slice 1 behavior

### Out Of Scope
