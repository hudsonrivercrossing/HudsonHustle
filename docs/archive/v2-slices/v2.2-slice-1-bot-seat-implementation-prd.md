# V2.2 Slice 1 Implementation PRD

## Product Summary

This document scopes the first implementation slice of `v2.2` system player work.

Slice 1 proves that Hudson Hustle can run a built-in server-owned `bot` seat through the existing authoritative multiplayer architecture without yet exposing the full feature in normal setup UX.

## Problem

The `v2.2` system-player direction is agreed, but the riskiest part is still unproven:
- can a seat be controlled by the server rather than a human client
- without bypassing authoritative validation
- while preserving hidden-information boundaries
- and without regressing human-only rooms

If that architectural step is not proven first, later setup UX and bot-quality work rest on unstable ground.

## Goals

- Introduce the minimum server-owned `bot` seat path.
- Prove that the bot can take legal turns through the same authoritative room/game flow as a human seat.
- Keep the first path deterministic and testable.
- Preserve current human-only room behavior.
- Avoid prematurely expanding into bot UX, difficulty settings, or external agents.

## Non-Goals

- No normal room-setup controller picker yet.
- No multiple bot-seat setup UX yet.
- No difficulty tiers.
- No advanced strategy layer beyond “legal and deterministic baseline.”
- No player-facing explanation UI.
- No external agent/plugin architecture.

## User Stories

1. As an engineer, I want one server-owned `bot` seat path so I can prove the controller architecture before exposing it in normal setup.
2. As a future product designer, I want the first bot path to use the same seat and action contracts as a human so later mixed-controller rooms do not require a rewrite.
3. As a maintainer, I want strong deterministic tests around the first bot turn loop so later bot improvements do not destabilize room progression.

## Solution

Slice 1 should add the minimum internal bot-seat infrastructure:
- a seat/controller representation that can distinguish a `bot` seat from a human seat
- a server-owned turn path that can decide and submit a legal action for that bot seat
- a deterministic baseline decision adapter
- test coverage that proves the bot:
  - only acts when it controls the active seat
  - submits legal actions through normal server flow
  - completes turn progression correctly

This slice may use a narrow, internal-only way to create or exercise a bot seat, but it should not require the final normal-setup UX yet.

## Scope

### In Scope

- server-side seat/controller support for one internal bot seat path
- minimum room-service changes needed to let the server act for a bot seat
- deterministic baseline action selection
- server tests for:
  - seat/controller lifecycle
  - legal bot action submission
  - turn completion and handoff
  - hidden-information boundaries
- any minimal internal/dev-only trigger needed to exercise the path

### Out Of Scope

- normal product-facing room setup for bot seats
- multiple bot-seat configuration UX
- difficulty selection
- advanced strategy tuning
- `human+agent` behavior
- external controllers

## Implementation Decisions

- The first bot path should remain server-owned.
- The bot must not mutate game state directly; it should submit normal legal action intents through the authoritative room/game path.
- The bot should observe only the same public/private view a seat should have.
- The initial decision adapter should be deterministic or seed-controlled.
- The first slice should optimize for architecture proof and testability, not player-facing polish.

## Testing Decisions

- Add deterministic server tests for:
  - creating or exercising a room with a bot-controlled seat
  - bot turn ownership
  - bot legal action progression
  - no regressions in human-only rooms
- Prefer behavior-first assertions:
  - active seat advances correctly
  - legal actions are applied
  - room status remains valid
- Do not rely on manual browser-only validation for the architectural proof.

## Open Questions

- What is the smallest internal trigger for creating/exercising a bot seat without prematurely locking final setup UX?
- Which legal decisions are simplest and safest for the first deterministic baseline: ticket confirmation, blind card draw, and a small claim subset?
- Should the baseline policy live entirely in `apps/server`, or should read-only evaluation helpers be added to shared code immediately?
