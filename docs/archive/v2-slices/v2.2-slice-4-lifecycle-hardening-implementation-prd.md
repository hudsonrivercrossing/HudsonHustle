# V2.2 Slice 4 Implementation PRD

## Product Summary

This document scopes the fourth implementation slice of `v2.2` system player work.

Slice 4 turns the first public mixed human/bot rooms from functionally correct into operationally trustworthy by hardening timer, reconnect, and mixed-room lifecycle behavior.

## Current Outcome

Slice 4 is now implemented:
- timed human turns can hand off through server-owned bot turns without stalling
- persisted mixed rooms resume correctly when a bot seat is active
- reconnect and disconnect semantics remain explicitly human-only while mixed-room public state stays stable
- regression coverage now includes timed mixed-room and reconnect lifecycle paths

## Problem

Slices 1 through 3 proved the controller model, exposed `bot` seats in normal setup, and strengthened the baseline bot policy. But mixed rooms can still fail in operational edges if lifecycle behavior is only implicitly correct:
- a timed human timeout can hand off into a `bot` turn without immediately resuming server-controlled play
- a persisted room can reload with a `bot` seat active and stay stuck until another event arrives
- reconnect and disconnect behavior in mixed rooms can drift away from clear human-only semantics

That would make the milestone look feature-complete while still being brittle in the real world.

## Goals

- Preserve normal per-turn timer behavior when control passes between human and server-owned seats.
- Ensure persisted or reloaded mixed rooms resume correctly without waiting on impossible client input from a `bot` seat.
- Keep reconnect semantics clear:
  - human seats reconnect
  - `bot` seats do not reconnect
  - mixed rooms remain understandable while reconnects happen
- Prove these paths with deterministic server coverage and browser regression where it matters.

## Non-Goals

- No new bot-strength heuristics.
- No difficulty tiers.
- No external agents or `human+agent` UX.
- No redesign of multiplayer setup or lobby hierarchy.

## User Stories

1. As a player in a timed mixed room, I want the game to keep moving correctly after a human timeout even if the next turn belongs to a `bot`.
2. As a reconnecting human player, I want a mixed room to restore sensibly without bot seats pretending to be reconnectable.
3. As a maintainer, I want mixed-room lifecycle behavior to stay deterministic and recoverable across persistence and reconnect.

## Solution

Slice 4 should harden the operational lifecycle around the existing `bot` seat feature:
- when a timed human turn expires and the next seat is server-owned, the server should immediately continue the bot turn loop
- when an active mixed room is restored from persistence, server-owned active seats should resume without waiting on a human request path
- reconnect and disconnect semantics should stay explicitly human-only while the mixed room continues to broadcast correct public state
- regression coverage should make these behaviors hard to break accidentally

## Scope

### In Scope

- timer handoff from human timeout into server-owned bot turns
- reload/resume behavior for persisted active mixed rooms
- reconnect/disconnect lifecycle hardening for mixed rooms
- deterministic server tests for timed mixed-room behavior
- browser regression for reconnect/timer behavior where user-visible
- docs sync for lifecycle and controller semantics

### Out Of Scope

- stronger bot policy heuristics
- difficulty tiers
- player-facing lifecycle redesign beyond clarity fixes
- external-agent protocol work

## Implementation Decisions

- Server-owned active seats must resume through the same authoritative server loop used in earlier slices.
- Lifecycle hardening should prefer explicit resume logic over UI-only guards.
- Human reconnect remains secret-based and client-owned.
- `bot` seats remain server-owned and non-reconnectable.
- Mixed-room hardening should reuse the current room model instead of introducing a second lifecycle path for bots.

## Testing Decisions

- Add deterministic server tests for:
  - timeout handoff into bot turns
  - persisted mixed-room resume when a bot seat is active
  - reconnect/disconnect semantics in mixed rooms
- Add browser regression for at least one visible timed mixed-room and reconnect flow.
- Keep existing Slice 1-3 coverage green.
