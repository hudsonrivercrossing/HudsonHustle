# V2.2 System Player Plan

## Summary
- This plan implements [V2.2 System Player PRD](/Users/djfan/Workspace/HudsonHustle/docs/product/v2/v2.2-system-player-prd.md).
- Milestone target: the first bounded `v2.2` release after `v2.1` freeze.
- Product goal: ship a built-in server-owned deterministic `bot` seat that works through the normal authoritative multiplayer stack and can appear in normal room setup.
- Current status: Slices 1 through 5 are complete, and the first public `v2.2` system-player milestone is now frozen.

## Scope Boundary

### In Scope
- add `bot` as the first shipped non-human controller type
- support mixed human/bot seat compositions up to room size
- keep the bot server-owned and routed through normal authoritative room/game flows
- add the minimum setup UX for assigning one or more bot seats
- ship one competent baseline deterministic policy
- add deterministic server coverage plus browser smoke for setup changes
- align `v2` docs with the first system-player milestone

### Out Of Scope
- external model-driven agents
- user-provided autonomous agents or plugin protocols
- `human+agent` assistance UX
- difficulty tiers such as `easy` / `medium` / `hard`
- matchmaking, ranking, or solo campaign framing
- redesign of core rules, scoring, or map/config balance

## Slice Order

### Slice 1
- Objective:
  - Prove the seat/controller architecture with one internal server-owned `bot` seat in a non-public or minimally exposed path.
- Why first:
  - This is the riskiest architectural step.
  - It proves that a seat can be controlled by the server without bypassing authoritative game rules.
- Likely areas touched:
  - `apps/server/src/room-service.ts`
  - `apps/server/src/app.ts`
  - multiplayer room/seat types and snapshots
  - shared controller/seat typing in `packages/game-core` only if required by public contracts
- Validation:
  - deterministic server tests for seat creation, turn ownership, legal action submission, and turn completion
  - verify human-only rooms still behave exactly as before
- Guardrail:
  - no public UI commitment yet beyond the minimum needed to exercise the path
  - no heuristic sophistication beyond “legal and deterministic”

### Slice 2
- Objective:
  - Expose `bot` seat selection through normal room setup and allow mixed human/bot rooms up to room size.
- Why here:
  - Once the architecture is proven, the next highest-value step is making the feature real in normal product flow.
- Likely areas touched:
  - `apps/web/src/App.tsx`
  - `apps/web/src/components/MultiplayerSetupScreen.tsx`
  - lobby/setup seat rendering and related form/state logic
  - server room create/join/setup request handling
- Validation:
  - browser tests covering:
    - create room with one bot
    - create room with multiple bots
    - mixed room proceeds to start correctly
  - verify setup remains readable and does not regress `v2.1` shell hierarchy
- Guardrail:
  - keep setup UX minimal and seat-oriented
  - do not introduce a large new controller-management surface

### Slice 3
- Objective:
  - Replace a purely minimal legal bot with a competent baseline deterministic policy informed by current Hudson Hustle winning heuristics.
- Why here:
  - A legal-but-bad bot proves architecture, but a competent baseline is required before the feature is credible in normal setup.
- Likely areas touched:
  - new server-side bot decision module(s)
  - `packages/game-core` read-only helpers if needed for state evaluation
  - `.codex/skills/how-to-win-hudson-hustle/SKILL.md` only if terminology or heuristics references need follow-up
- Validation:
  - deterministic behavior tests against fixed states
  - regression tests for:
    - avoids obviously illegal or self-destructive choices
    - can complete full turn loops
    - handles ticket selection, draw decisions, and route claims within legal constraints
- Guardrail:
  - do not add difficulty levels yet
  - do not move strategy logic into React/UI code

### Slice 4
- Objective:
  - Harden timing, reconnect, and mixed-room lifecycle behavior for bot seats.
- Why here:
  - This is where architecture becomes operationally trustworthy instead of only functionally correct.
- Likely areas touched:
  - server timer scheduling
  - room lifecycle/reconnect logic
  - multiplayer browser tests
  - docs for room/controller behavior
- Validation:
  - server tests for bot turns under timed rooms
  - browser smoke for mixed human/bot room progression
  - verify reconnect semantics stay sensible for human seats and irrelevant for server-owned seats
- Guardrail:
  - do not broaden into external-agent protocol work
  - do not hide lifecycle bugs behind UI-only guards

### Slice 5
- Objective:
  - Freeze the first `v2.2` system-player milestone and align docs.
- Why here:
  - The first public/normal-flow bot milestone needs stable terminology and clear follow-on boundaries.
- Likely areas touched:
  - `docs/product/tech-spec.md`
  - `docs/product/v2/v2-mvp-architecture.md`
  - `docs/product/v2/v2.2-system-agent-architecture.md`
  - possibly `docs/gameplay/player-guide.md` if player-facing setup language changes
- Validation:
  - one findings-first review pass
  - docs check for terminology consistency:
    - `bot`
    - `agent`
    - `human+agent`
    - `seat`
    - `controller`
- Guardrail:
  - no late design-system side quests
  - no speculative `v2.3` architecture creep inside the freeze pass

## Risks
- hidden-information leakage if the bot sees more than a seat should
- architecture drift if bot action submission bypasses the normal authoritative path
- mixed-room setup complexity making the normal lobby heavier than `v2.1`
- a baseline bot that is technically legal but too weak to be credible
- timer or reconnect edge cases behaving differently once server-owned turns are active
- terminology drift between `bot`, `agent`, and `human+agent`
