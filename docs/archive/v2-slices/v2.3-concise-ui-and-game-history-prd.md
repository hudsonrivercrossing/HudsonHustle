# V2.3 Concise UI And Game History PRD

## Product Summary

`v2.3` is a consolidation milestone after the `v2.2` system-player release.

It has two tightly scoped goals:
- make the shipped online multiplayer UI more concise, cleaner, and more visually coherent
- persist a complete online multiplayer match history so every finished game can later be reviewed move-by-move and round-by-round

This milestone is intentionally not a replay milestone, not an AI-highlights milestone, and not a larger UX redesign milestone.

## Problem

`v2.2` now ships a real online multiplayer product with mixed human/`bot` rooms, authoritative backend behavior, reconnect, timers, and production deployment.

But two gaps remain before the next broader UX or agent step:

1. the live UI still carries too much historical drift
- some layout areas are heavier than they need to be
- some visual treatments still reflect older font/color choices
- some information hierarchy is noisier than the current product direction wants

2. finished games still do not leave behind a canonical match record
- the backend can resume active rooms, but it does not yet preserve a full readable history of each finished online match
- future replay, post-game highlights, and strategy review would be forced to reconstruct too much from incomplete artifacts
- future bot-skill generation from real games would not have a trustworthy source record

That means the product is playable, but not yet concise enough on the surface or instrumented enough underneath.

## Goals

- Simplify the shipped online multiplayer UI without reopening core gameplay or broad information architecture work.
- Remove low-value chrome and legacy visual drift from the live gameplay shell.
- Persist a durable online multiplayer match history for every new completed game.
- Record the game clearly enough to review it move-by-move and round-by-round, similar to a chess game record.
- Use a pragmatic history model that is stable enough for future replay, highlights, and bot-learning work without over-engineering the runtime now.

## Non-Goals

- No replay UI in `v2.3`.
- No AI-generated post-game highlights in `v2.3`.
- No bot-skill generation from saved games in `v2.3`.
- No local pass-and-play history capture in `v2.3`.
- No large `v2.4` UX redesign work.
- No `v3` controller work such as `human+agent` or full autonomous `agent` seats.
- No gameplay rule, scoring, map, or balance redesign unless required to correctly log history.

## User Stories

1. As a player, I want the live online game interface to feel cleaner and easier to scan so the board state stays central.
2. As a maintainer, I want every completed online game to leave behind a trustworthy move history so I can inspect what actually happened.
3. As a future product builder, I want saved match history to be detailed enough for later replay, highlight, and strategy features without redoing the persistence model.

## Solution

Hudson Hustle `v2.3` should combine one visible product cleanup and one backend foundation:

1. concise shell cleanup
- reduce or remove low-value UI elements
- tighten layout and hierarchy
- align lingering font/color/style drift with the current design system direction

2. game-history foundation
- persist a canonical ordered match history for online multiplayer only
- use `action log + checkpoint snapshots`
- keep the log detailed enough to reconstruct a match move-by-move
- keep the first retrieval surface internal or otherwise low-prominence

The milestone should remain deliberately narrow:
- cleaner shipped UI now
- better instrumentation now
- replay/highlights/bot-learning later

## Product Decisions

The first `v2.3` PRD locks these decisions:

1. History scope is online multiplayer only.
- `v2.3` should not expand into local pass-and-play history capture.
- The authoritative online backend is the right first foundation because it already owns the canonical game state.

2. The first history model should use `action log + checkpoint snapshots`.
- Do not start with full snapshot-per-move persistence.
- Do not refactor the entire backend into pure event sourcing.
- The model should be detailed enough to replay the game logically, but operationally simple enough to ship as a foundation slice.

3. The required history fidelity is move-by-move and round-by-round.
- A future consumer should be able to understand the game sequence like a chess record.
- The stored history should not collapse a whole game down to final score only or broad turn summaries.

4. Replay UI is not part of `v2.3`.
- `v2.3` prepares replay data, not the replay product surface.

5. Bot-skill generation from saved games belongs to `v2.4`, not `v2.3`.
- `v2.3` only lays the data foundation for that future work.

6. UI work in `v2.3` should be reductive, not expansive.
- Remove clutter.
- Tighten hierarchy.
- Do not add new large interaction surfaces while calling the milestone “simplification.”

## Scope

### In Scope

- simplify the live online multiplayer shell
- remove or reduce low-value gameplay chrome
- align lingering typography/color/layout drift
- define a canonical online multiplayer game-history contract
- persist move-by-move match history for new completed online games
- add sparse checkpoint snapshots for reconstruction
- keep the first history retrieval path internal or low-prominence
- add deterministic validation for history completeness and UI smoke confidence

### Out Of Scope

- replay viewer
- player-facing post-game narrative or highlight generation
- bot learning or model training
- local game history
- broad onboarding redesign
- board/map redesign

## Implementation Decisions

- `v2.3` should build on the existing authoritative multiplayer backend and current persistence model.
- The current latest-room snapshot path should remain the runtime restore path.
- The current lightweight `game_events` storage should evolve into a canonical gameplay history log instead of being replaced by a wholly separate logging stack.
- A new immutable checkpoint snapshot table should be added for historical reconstruction.
- Normal history events should avoid storing full private state directly in each event payload.
- UI simplification should preserve gameplay clarity and should not silently alter game rules or action semantics.

## Testing Decisions

- Add deterministic server tests for complete online multiplayer history persistence.
- Ensure mixed human/`bot` rooms are included in history validation so the contract is not implicitly human-only.
- Add browser smoke for setup, lobby, gameplay, and endgame after UI simplification.
- Keep existing multiplayer runtime validation green while introducing history persistence.
