# V2.2 Slice 2 Implementation PRD

## Product Summary

This document scopes the second implementation slice of `v2.2` system player work.

Slice 2 takes the internal bot-seat architecture proof from Slice 1 and exposes it through normal multiplayer room setup so a room can include any mix of human and bot seats up to room size.

## Problem

Slice 1 proved the server-owned `bot` seat path, but the feature is not yet available through the normal product flow.

Without Slice 2:
- the system player remains an internal engineering capability rather than a real `v2.2` feature
- mixed human/bot rooms cannot be created through normal setup
- the architecture has not yet been proven against the actual room-creation, lobby, and start-game flow

The next milestone must make `bot` seats available in normal setup without making the `v2.1` multiplayer flow feel significantly heavier.

## Goals

- Expose `bot` seat selection through normal multiplayer room setup.
- Support any number of bot seats up to room size.
- Keep setup seat-oriented and readable.
- Let mixed human/bot rooms reach game start through the normal room lifecycle.
- Preserve the `v2.1` shell hierarchy and human-only room behavior.

## Non-Goals

- No difficulty selection or strategy tuning in this slice.
- No reconnect semantics for bot seats beyond “not applicable.”
- No external agents or `human+agent` UI.
- No new room-management dashboard or advanced controller editor.
- No lifecycle hardening beyond the minimum needed for normal setup/start.

## User Stories

1. As a host, I want to add one or more `bot` seats while creating a room so I can start a mixed table without recruiting a full set of humans.
2. As a human player joining a mixed room, I want the lobby to show which seats are human and which are `bot` seats so the room remains understandable.
3. As a maintainer, I want normal setup and start flow coverage for mixed rooms so the first public bot feature does not regress the stable `v2.1` multiplayer path.

## Solution

Slice 2 should expose controller selection in the normal room setup flow using the minimum new UI and server surface needed for seat-oriented mixed rooms.

The slice should add:
- setup state that can represent `open`, `human`, and `bot` seat intent up to room size
- server room-create/start handling that accepts bot seats as first-class seat/controller choices
- setup and lobby rendering that clearly distinguishes human and bot seats without introducing a heavy controller-management surface
- browser and server coverage for one-bot and multi-bot rooms proceeding through normal start flow

The product shape should remain:
- simple room setup
- seat-oriented
- visually consistent with `v2.1`

It should not become:
- a general controller orchestration UI
- a difficulty-selection or bot-configuration panel

## Scope

### In Scope

- normal setup support for choosing human or `bot` seat occupancy up to room size
- mixed human/bot room creation in the normal multiplayer setup flow
- lobby/setup rendering that identifies `bot` seats clearly
- start-game flow for mixed rooms through the authoritative backend
- validation for:
  - one-bot rooms
  - multiple-bot rooms
  - human-only regression safety
- docs updates for player-facing or room-facing bot setup language if needed

### Out Of Scope

- bot difficulty tiers
- advanced bot customization
- timer/reconnect hardening beyond existing normal-room behavior
- coach, assistant, or `human+agent` concepts
- broad lobby redesign

## Implementation Decisions

- Setup should stay seat-oriented, not player-type dashboard-oriented.
- The host should be able to assign any seat as a `bot` seat until room size is reached.
- A `bot` seat should be visible as a normal seat with a server-owned controller, not as a hidden room flag.
- The setup UI should use the existing shell/system language and stay lighter than a generic admin form.
- Human-only room creation should remain as close as possible to `v2.1`.
- The slice should reuse Slice 1 controller infrastructure rather than adding a second path.

## Testing Decisions

- Add browser coverage for:
  - create room with one bot
  - create room with multiple bots
  - mixed room proceeds to game start correctly
- Add targeted server coverage for:
  - room creation with mixed controller selections
  - lobby/start state for bot seats
  - no regressions in human-only room setup
- Keep assertions behavior-first:
  - room reaches the right status
  - seats are represented correctly
  - bot seats participate without bypassing the normal lifecycle

## Open Questions

- What is the lightest seat picker or seat-row affordance that makes `bot` assignment clear without making setup feel like a control panel?
- Should a seat default to `open` and then be switched to `bot`, or should normal room size selection materialize seat rows that are then assigned?
- What is the minimum player-facing copy needed so `bot` is clear while `agent` and `human+agent` remain reserved for later work?
